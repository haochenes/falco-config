# Falco 交叉编译 — 开发者手册

本文档面向在 **TI J721E / TDA4VM** 等 ARM64 板子上部署 Falco 的开发者，说明编译环境、两种驱动模式的编译与部署方法。

---

## 1. 编译器版本与硬件平台

### 1.1 目标硬件平台

| 项目 | 说明 |
|------|------|
| **架构** | ARM64（aarch64） |
| **典型平台** | TI J721E、TDA4VM（TI Processor SDK Linux ADAS） |
| **内核** | 与板端运行内核一致（如 6.1.x-ti-*），用于编译 falco.ko 时需指定内核源码路径 |

### 1.2 主机编译环境

| 项目 | 说明 |
|------|------|
| **交叉编译器** | ARM GNU Toolchain（aarch64-none-linux-gnu-gcc/g++） |
| **推荐版本** | 11.3.rel1（如 `arm-gnu-toolchain-11.3.rel1-x86_64-aarch64-none-linux-gnu`） |
| **目标三元组** | `aarch64-none-linux-gnu` |
| **Sysroot** | TI Processor SDK Linux ADAS J721E EVM 提供的目标系统根目录（库与头文件） |

### 1.3 配置文件 build.cfg 中的关键路径

在 `cross_compile/build.cfg` 中需正确配置（路径需与本地环境一致）：

```ini
# Sysroot：目标板对应的根文件系统（库、头文件）
SYSROOT="/opt/ti-processor-sdk-linux-adas-j721e-evm-09_02_00_05/linux-devkit/sysroots/aarch64-oe-linux"

# 交叉编译器安装目录
CROSS_COMPILE_PREFIX="/opt/cross-compile/arm-gnu-toolchain-11.3.rel1-x86_64-aarch64-none-linux-gnu/"
CROSS_COMPILE_TRIPLE="aarch64-none-linux-gnu"

# 仅 kmod 模式需要：板端运行内核的源码树（用于编译 falco.ko）
LINUX_KERNEL_SRC="/home/carlos/work/tda4-bsp/bsp/ti-linux-kernel"
```

### 1.4 主机必备工具

- **CMake** 3.16+
- **Git**、**Make**、**wget**
- **kmod 模式**：无额外要求。
- **Modern eBPF 模式**（若在本机尝试）：需 **clang**、**llvm**；可选 **libelf-dev**、**zlib1g-dev**、**bpftool**（用于 BPF 编译）。  
  注意：当前交叉编译 modern eBPF 会因 `events_dimensions_generator` 限制而失败，板端推荐使用 **kmod 模式**。

## 2. 两种模式的编译方法

Falco 在板上有两种驱动方式，对应两种编译方式：

- **kmod 模式**：板端加载内核模块 `falco.ko`，Falco 使用 `engine.kind: kmod`。**当前交叉编译推荐方式。**
- **Modern eBPF 模式**：板端无需 .ko，使用内核 eBPF（需内核支持 BTF，一般 5.8+）。交叉编译该模式目前受 falcosecurity-libs 限制易失败，本手册仍说明用法，供本机或未来支持时使用。

### 2.1 模式一：kmod 模式（推荐用于板端）

**特点**：产出 `falco` 可执行文件 + `falco.ko` 内核模块；板端通过 `insmod falco.ko` 后启动 Falco。

**步骤一：配置 build.cfg**

```ini
BUILD_MODERN_BPF="OFF"
BUILD_KMOD="ON"
MINIMAL_BUILD="ON"
# 必须指向与板端内核一致的内核源码树，且已执行过 make 或 make modules_prepare
LINUX_KERNEL_SRC="/path/to/your/ti-linux-kernel"
```

**步骤二：执行编译**

```bash
cd cross_compile
chmod +x build_falco.sh
./build_falco.sh all
```

**步骤三：查看产物**

- 可执行文件：`cross_compile/install/bin/falco`
- 内核模块：`cross_compile/install/share/falco/falco.ko`
- 配置与规则：`install/etc/falco/`、`install/share/falco/`

常用子命令：

- `./build_falco.sh configure` — 仅配置
- `./build_falco.sh build` — 仅编译
- `./build_falco.sh install` — 仅安装到 `install/`
- `./build_falco.sh clean` — 清理构建目录
- `./build_falco.sh help` — 查看帮助

### 2.2 模式二：Modern eBPF 模式（无 .ko，板端需 BTF 内核）

**特点**：仅产出 `falco` 可执行文件；板端配置 `engine.kind: modern_ebpf`，无需编译或加载 falco.ko。  
**限制**：交叉编译时 falcosecurity-libs 需要 host 端 `events_dimensions_generator`，当前交叉构建会失败，仅说明用法。

**方式 A：使用主脚本 build_falco.sh（本机或未来支持时）**

在 `build.cfg` 中：

```ini
BUILD_MODERN_BPF="ON"
BUILD_KMOD="OFF"
MINIMAL_BUILD="OFF"
```

主机需安装 clang、llvm 等（见 1.4），然后：

```bash
cd cross_compile
./build_falco.sh all
```

产物在 `install/bin/falco`，无 `falco.ko`。

**方式 B：使用独立脚本 build_falco_ebpf.sh**

- 使用**独立目录** `build_ebpf/`、`install_ebpf/`，不覆盖 `build_falco.sh` 的 `build/`、`install/`。
- 仅开启 modern eBPF，不编内核模块。

```bash
cd cross_compile
./build_falco_ebpf.sh all
```

产物在 `install_ebpf/bin/falco`。  
若出现 **events_dimensions_generator: not found**，属当前交叉编译限制，请改用 **kmod 模式** 编译并在板端使用 `engine.kind: kmod`。

## 3. 两种模式的部署与使用方法

部署前请先完成对应模式的编译，并确保板子与主机网络互通（如 SSH）。

### 3.1 板端路径与 board_test 配置

部署脚本使用 `board_test/board.cfg`，默认将 Falco 部署到：

- 可执行文件：`/usr/local/bin/falco`
- 配置：`/etc/falco/`
- 插件与规则：`/usr/share/falco/`
- 内核模块（仅 kmod）：`/usr/share/falco/falco.ko`

`board.cfg` 中的 `FALCO_INSTALL_DIR` 指向**本机**的编译产物目录：

- **kmod 模式**：`FALCO_INSTALL_DIR="cross_compile/install"`（使用 `build_falco.sh` 的产出）
- **Modern eBPF 模式**（若用 build_falco_ebpf.sh）：需改为 `FALCO_INSTALL_DIR="cross_compile/install_ebpf"`，或先手动将 `install_ebpf` 内容拷贝到 `install` 再部署。

### 3.2 kmod 模式：部署与使用

**1）部署到板子**

在仓库根目录执行（会读取 `board_test/board.cfg` 的 IP、用户等）：

```bash
cd board_test
./deploy_to_board.sh
```

脚本会拷贝：

- `cross_compile/install/bin/falco` → 板端 `/usr/local/bin/falco`
- `install/etc/falco/` → 板端 `/etc/falco/`
- `install/share/falco/`（含 `falco.ko`、插件、规则）→ 板端 `/usr/share/falco/`
- 可选：`load-falco-ko.sh`、`falco.service` 等到 `/opt/falco-test/`

**2）板端加载模块并启动 Falco**

```bash
# 加载内核模块（每次上电或更新 .ko 后需要）
sudo insmod /usr/share/falco/falco.ko
# 若之前加载过旧版本，先卸载再加载：
# sudo rmmod falco; sudo insmod /usr/share/falco/falco.ko

# 前台运行
/usr/local/bin/falco -c /etc/falco/falco.yaml
```

或使用 systemd（若已部署并安装服务单元）：

```bash
# 将 falco.service 拷到系统目录并重载
sudo cp /opt/falco-test/falco.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl start falco
```

**3）板端配置**

保证 Falco 使用 kmod 引擎，例如在 `/etc/falco/falco.yaml` 或 `config.d/` 下配置：

```yaml
engine:
  kind: kmod
```

部署脚本在带容器插件时会使用 `falco.container_plugin.board.yaml` 等，一般已包含或兼容 kmod。

### 3.3 Modern eBPF 模式：部署与使用（当交叉编译可用时）

**1）部署**

若使用 `build_falco_ebpf.sh` 且希望用 `deploy_to_board.sh`，可先将 modern eBPF 产出当作“当前 install”再部署，例如：

```bash
# 临时让部署使用 install_ebpf（或在 board.cfg 中设 FALCO_INSTALL_DIR="cross_compile/install_ebpf"）
export FALCO_INSTALL_DIR="cross_compile/install_ebpf"
cd board_test
./deploy_to_board.sh
```

或手动拷贝：

```bash
scp install_ebpf/bin/falco root@<板子IP>:/usr/local/bin/
scp -r install_ebpf/etc/falco root@<板子IP>:/etc/
scp -r install_ebpf/share/falco root@<板子IP>:/usr/share/
```

**2）板端配置**

在 `/etc/falco/config.d/` 下使用 modern eBPF 配置，例如拷贝仓库中：

- `board_test/config/falco.modern_bpf.board.yaml`

或自行设置：

```yaml
engine:
  kind: modern_ebpf
```

**3）板端启动**

```bash
/usr/local/bin/falco -c /etc/falco/falco.yaml
```

无需 `insmod`；内核需支持 BTF（如 5.8+）。

### 3.4 快速对照表

| 项目 | kmod 模式 | Modern eBPF 模式 |
|------|-----------|-------------------|
| 编译脚本 | `build_falco.sh`（BUILD_KMOD=ON） | `build_falco_ebpf.sh` 或 build_falco.sh（BUILD_MODERN_BPF=ON） |
| 产物目录 | `install/` | `install_ebpf/` 或 `install/` |
| 是否需 falco.ko | 是，需部署并 insmod | 否 |
| 板端 engine.kind | `kmod` | `modern_ebpf` |
| 板端内核要求 | 与 LINUX_KERNEL_SRC 一致即可 | 需支持 BTF（通常 5.8+） |
| 交叉编译状态 | 推荐，已验证 | 当前受 events_dimensions_generator 限制 |

## 4. QEMU 方案说明与多板卡（TDA4 / S32G）风险

官方文档中有时会建议用 **QEMU user-mode + Docker 容器** 在“仿真 aarch64 环境”里直接编译，例如：

```bash
# 1. 安装仿真器
sudo apt-get install qemu-user-static
# 2. 注册 binfmt_misc（让内核识别 ARM 二进制）
docker run --rm --privileged multiarch/qemu-user-static --reset -p yes
# 3. 在 aarch64 的容器镜像中进行编译
docker run -it -v $(pwd):/workspace arm64v8/ubuntu:22.04 /bin/bash
# 容器内再执行 Falco 的编译
```

这种方式可以规避“交叉编译时 host 工具（如 events_dimensions_generator）无法运行”的问题，但会引入以下风险和限制，**在需要同时支持 TDA4 与 S32G399A 时尤其要注意**。

### 4.1 主要风险

| 风险 | 说明 |
|------|------|
| **ABI / 运行库不一致** | 在 `arm64v8/ubuntu:22.04` 里编出的二进制是**针对 Ubuntu 22.04 arm64** 的：链接的是该镜像里的 glibc、libstdc++、系统库。而 **TDA4** 跑的是 TI Processor SDK（多为 Yocto/OE），**S32G399A** 跑的是 NXP SDK，各自 glibc 版本、库路径、可选依赖都可能不同。把“为 Ubuntu 编的”二进制直接拷到 TDA4/S32G 上，可能出现：<br>• 运行时报找不到符号（glibc 版本过新）、<br>• 依赖库缺失或版本不匹配、<br>• 偶发 ABI 兼容问题。 |
| **并非“为当前板子”构建** | 交叉编译 + sysroot 的方式是**明确用某块板子对应的 sysroot**（TI 或 NXP 提供的根文件系统）链接，得到的二进制和该 BSP 的 libc/ABI 一致。QEMU + 通用 Ubuntu 镜像是“为 Ubuntu arm64”构建，不是为 TDA4 或 S32G 的 BSP。 |
| **编译速度** | QEMU user-mode 下每条指令都是模拟执行，编译会明显慢于本机交叉编译（可能慢一个数量级），CI/日常迭代成本高。 |
| **多板卡需多套环境** | 若坚持用 QEMU：TDA4 与 S32G 应使用**各自 BSP 对应的根文件系统/镜像**在容器里编译（例如基于 TI SDK 的 rootfs 做 TDA4 镜像、基于 NXP SDK 的 rootfs 做 S32G 镜像），而不是用一个通用 Ubuntu arm64 通吃，否则 ABI 问题更不可控。 |

### 4.2 为何多板卡（TDA4 + S32G）更推荐交叉编译 + sysroot

- **TDA4**：用 TI 提供的 sysroot 和（可选）内核源码，`build.cfg` 里 `SYSROOT`、`LINUX_KERNEL_SRC` 指向 TDA4 的 BSP；得到的是为 **TI 该版 BSP** 定制的 Falco。
- **S32G399A**：换用 NXP 提供的 sysroot 和内核源码，同一套 `build_falco.sh`，只改 `build.cfg`（以及必要时为 S32G 单独建一份配置或脚本）；得到的是为 **NXP 该版 BSP** 定制的 Falco。
- 两套产物**各自与对应板子的 glibc/内核 ABI 一致**，无需在板子上再装额外运行库，也不依赖“Ubuntu 与 BSP 刚好兼容”。

### 4.3 若仍采用 QEMU 时的建议

- **不要用通用 Ubuntu arm64 镜像作为“通吃”构建环境**；至少为 TDA4、S32G 各准备一个与 BSP 一致的 rootfs/镜像（可从 SDK 或 Yocto 构建出的 rootfs 制作 Docker 镜像），在对应容器里编译。
- 在目标板子上做**冒烟测试**：编译产物拷贝到 TDA4 与 S32G 后，执行 `falco -v`、实际加载规则并跑一小段，确认无动态链接错误和段错误。
- 若仅做验证或一次性构建，可接受“可能偶发 glibc 兼容问题”，再考虑在 Ubuntu arm64 容器里编译并到多块板子上试跑；**长期支持多板卡时，仍建议以交叉编译 + 每板专用 sysroot 为主方案**。

## 5. 附录：目录结构速览

```
cross_compile/
├── build.cfg              # 编译配置（编译器、sysroot、kmod/eBPF 开关）
├── build_falco.sh          # 主编译脚本（kmod 或可选 modern eBPF）
├── build_falco_ebpf.sh     # 仅 modern eBPF 编译（独立 build_ebpf/、install_ebpf/）
├── toolchain-aarch64.cmake
├── 开发者手册.md          # 本文档
├── README.md
├── src/                   # 源码（脚本自动拉取）
├── build/                 # kmod 构建目录
├── install/               # kmod 模式安装输出（供部署）
├── build_ebpf/            # 仅 build_falco_ebpf.sh 使用
└── install_ebpf/          # 仅 build_falco_ebpf.sh 使用
```

部署与测试脚本在仓库的 **board_test/** 下（如 `deploy_to_board.sh`、`run_tests_remote.sh`），详见 board_test 目录内说明。
